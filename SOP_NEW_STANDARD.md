# 新標準匯入與版本更新 SOP (Standard Operating Procedure)

這份 SOP 提供給 Antigravity (AI 助手) 與開發者，當未來有新的電池標準文件需要匯入系統時，請嚴格遵守以下標準化作業流程。

## 1. 觸發條件與檔案放置
當使用者表示有新的標準需要處理時，使用者會將原始文件（如 PDF 或 Markdown 檔）放入指定的資料夾（例如：`data/raw/` 或 `data/new_standards/`）。
AI 接收到處理指令後，即刻啟動本 SOP。

## 2. 內容結構化處理 (Schema v2.0 轉換)
1. **內容讀取與解析**：AI 需閱讀原始文件，並透過 OCR 或文本萃取技術理解標準內容。
2. **結構化為 JSON**：
   - 必須嚴格依照本系統目前的 `Schema v2.0` 格式進行轉換。
   - 確保提取出核心測試參數（如溫度、電壓、短路電阻、擠壓力等）與判定條件（不發火、不爆炸等）。
   - 確保豁免條件（Exemptions）被正確地嵌入對應的測試項目中。
   - 確保測試項目的 `normalized_id` 與 `taxonomy.json` 分類體系對齊。
3. **產出 JSON 檔案**：將結構化後的 JSON 檔案儲存至 `data/` 目錄中（例如：`data/GB38031-2026.json`）。

## 3. 版本關聯與資料庫註冊 (`catalog.json`)
1. **檢查是否為現有標準的新版**：
   - 透過比對標準代號（如 UN38.3, UL1642, GB31241），判斷這份新資料是「全新的標準」還是「現有標準的更新版」。
2. **全新標準註冊**：
   - 若為全新標準，直接在 `data/catalog.json` 寫入新的物件。
   - 預設 `is_latest: true`, `versions_behind: 0`。
3. **現有標準的更新版 (Version Update) 處理**：
   - **保留舊版**：絕對不可以刪除或覆寫舊版的 JSON 檔案與 `catalog.json` 中的舊紀錄。
   - **更新舊版狀態**：將舊版紀錄中的 `is_latest` 改為 `false`，並增加 `versions_behind` 的數值，同時更新 `latest_version` 欄位為新版的名稱。
   - **註冊新版**：在 `catalog.json` 中為新版建立一筆獨立的紀錄（給予不同的 `document_id`，例如從 `UN38.3` 變成 `UN38.3-Rev8`），並設定 `is_latest: true`。

## 4. 前端介面更新：新舊版本分頁切換 (Tab UI)
當發現匯入的是現有標準的「新版本」時，AI 必須連帶修改前端的介面設計：
1. **觸發條件**：當使用者在橫向對比視圖中選擇了包含多個版本的同一份標準（例如同時存在 UN38.3 Rev 6 與 Rev 8）。
2. **UI 實作要求**：
   - 不要在畫面上並排顯示多個冗長的同一標準欄位。
   - 請使用 **分頁 (Tabs) 或下拉選單 (Dropdown)** 的方式整合在同一個 `StandardColumn` 內。
   - 預設顯示「最新版本」的資料。
   - 使用者可以透過點擊 Tab (例如 `[ 2015版 ] [ 2023版 (最新) ]`)，在同一個卡片欄位內動態切換新舊測試條件，方便對比新舊版之間的差異。

## 5. 驗證與紀錄
- 執行 `npm run dev` 或 `npm run build` 確認 JSON 格式正確且不破壞前端渲染。
- 測試新舊版本分頁切換功能是否順暢。
- 在 `AGENTS.md` 中記錄本次匯入的標準名稱與版本更新重點。
